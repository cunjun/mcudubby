# v0.6 Roadmap

This document captures the current direction for `McuBuddy` after the core probe, ELF/DWARF, RTOS, RTT, flash, diagnosis, and GDB-server flows reached a stable, hardware-validated baseline.

The goal of `v0.6` is not to rebuild the core. It is to deepen the strongest parts of the tool and close the remaining gaps in:

- J-Link advanced tracing
- high-level diagnose orchestration
- AI-first examples
- release-quality operator documentation

---

## 1. Priority Summary

Recommended implementation order:

1. stronger `diagnose(symptom)` workflows
2. J-Link trace enhancements
3. AI examples and operator-facing examples
4. release-quality docs and demo materials

---

## 2. Theme A: J-Link Trace Enhancements

### Objective

Turn the J-Link backend from "fully usable for debug" into "the strongest backend for trace-oriented workflows".

### Current status

Already validated:

- connect / halt / reset / continue / step
- source-level debug
- flash erase / program / verify
- J-Link GDB server
- RTT
- DWT cycle counter

Implemented but not fully closed:

- `read_swo_log(...)` backend path
- firmware-side SWO configuration experiments

### Remaining goals

- complete board-level SWO text validation on hardware with a usable trace pin
- optional `read_itm_trace(...)`
- stronger stop-reason consistency for trace-heavy flows
- trace-oriented examples and validation scripts

### Suggested sequence

1. `read_cycle_counter()` was completed first
2. complete board-level SWO text capture validation
3. evaluate `ITM` support after SWO works on real hardware

### Definition of done

- unit tests
- real hardware validation
- `PROGRESS.md` updated
- `README.md` updated if the feature becomes externally useful

---

## 3. Theme B: Device Match / Patch System

### Objective

Reduce the amount of manual target guessing required by users and AI agents.

### Current status

Already implemented:

- chip name alias table
- backend-name normalization
- backend-specific connection fallback hints
- support-matrix metadata
- warnings and recovery guidance
- lightweight non-intrusive post-connect checks

### Remaining work

- fuzzy target matching for less exact user input
- more built-in target profiles
- optional richer board-specific patch behaviors when evidence justifies them

### Status assessment

This theme is already close to baseline-complete for the current hardware set. Remaining work is refinement, not foundation.

### Example outcomes

- `STM32F103C8T6` resolves cleanly to the expected backend target string
- common STM32 aliases no longer require trial-and-error
- problematic boards can auto-apply stable connect settings

---

## 4. Theme C: Richer RTOS Scenarios

### Objective

Keep RTOS validation realistic and representative of common blocked and wake-up paths.

### Current status

Already hardware-validated on the STM32L496 demo:

- task listing
- task context reads
- queue receive
- semaphore wait
- software timer / `Tmr Svc`
- mutex
- event group
- task notification
- ISR-to-task notification

### Remaining work

- package these scenarios more cleanly for examples and docs
- extend validation to additional boards or kernels only when it adds real coverage

### Status assessment

This theme is functionally complete for the current STM32L496 validation board. Most remaining work is packaging and broader coverage, not missing core capability.

### Why this still matters

These paths directly improve:

- evidence quality for RTOS debugging
- realism of AI debugging workflows

---

## 5. Theme D: Stronger `diagnose(symptom)` Workflows

### Objective

Make the high-level diagnosis entrypoint feel more like a reliable debugging orchestrator.

### Current state

`diagnose(symptom)` already routes to useful lower-level tools.

### Remaining gaps

- better symptom taxonomy
- more stable multi-step diagnosis patterns
- better automatic combination of:
  - probe state
  - ELF/DWARF
  - SVD
  - RTT
  - RTOS

### End-state

Given a user report such as:

- "board won't boot"
- "UART is silent"
- "task is stuck"

the AI should be able to follow a clearer default tool sequence with less manual prompting.

### Status assessment

This is currently the highest-value unfinished theme because it most directly affects whether `McuBuddy` feels like an AI-native debugger rather than a large tool catalog.

---

## 6. Theme E: AI Examples

### Objective

Teach AI agents how to use `McuBuddy` effectively through scenario-driven examples.

### Current status

`docs/ai-examples.md` exists and covers the initial common scenarios. It should now be treated
as the canonical home for compact, scenario-based examples rather than duplicating long example
flows in README files.

Current initial scenarios:

1. board will not boot
2. hardfault after reset
3. UART has no output
4. FreeRTOS task appears stuck
5. J-Link RTT validation

### Guideline

Each example should show:

- the user problem
- the recommended tool sequence
- the important returned fields
- how to interpret the evidence

### Remaining work

- add more real-board evidence snippets as hardware validation expands
- keep examples compact and link to `AI Playbook` for decision-tree detail
- avoid embedding machine-local paths or one-off demo files

## 7. Theme F: Release-Quality Documentation

### Objective

Make the project easier to adopt and evaluate from the outside.

### Current status

Already in place:

- `README.md`
- `README_CN.md`
- `docs/README.md`
- `AI Playbook`
- `AI Examples`
- `Support Matrix`
- `Tool Reference`
- `Quickstart`
- `Generic Board Workflow`
- `Board Validation Guide`

### Remaining documentation gaps

- short demo / GIF / video
- more real-board validation snapshots as new hardware paths are proven

### Why this matters

At this stage, product clarity is becoming as important as raw feature count.

## 8. Milestone Proposal

### v0.6-alpha

Focus:

- J-Link trace start
- initial device preflight foundation

Expected outcome:

- J-Link gains at least one additional trace-facing tool beyond RTT and CYCCNT
- target selection becomes less manual

### v0.6-beta

Focus:

- stronger `diagnose(symptom)` orchestration
- AI examples

Expected outcome:

- more reliable AI-directed diagnosis workflows
- easier onboarding for agents and operators

### v0.6

Focus:

- docs, examples, and release polish

Expected outcome:

- clearer external positioning
- easier onboarding for both humans and AI agents
